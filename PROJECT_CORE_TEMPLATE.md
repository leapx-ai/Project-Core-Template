# Project Core Template

这是一份通用项目核心文档模板。

它不是 PRD，不是商业计划书，也不是技术方案，而是一个项目的“判断系统”。

适用场景：

1. 新项目从 0 开始。
2. 项目仍在探索阶段，但希望保持方向清楚。
3. 项目涉及产品、内容、技术、设计、安全等多个维度。
4. 需要避免过早写太多文档。
5. 需要让协作者快速理解项目判断标准。

使用方式：

1. 复制本模板。
2. 删除不适用章节。
3. 用具体项目内容填充。
4. 保持短而清楚。
5. 后续优先更新 Project Core，而不是不断创建新文档。

## 使用建议

默认情况下，一个新项目只需要先创建一份文档：

1. `docs/PROJECT_CORE.md`  
   直接复制本模板并填充。它就是项目早期的核心蓝图，用于记录北极星、受众、核心闭环、不做清单、产品原则、技术原则和当前优先级。

不要一开始就为了“规范”创建很多文档。只有当项目真的出现对应需要时，再补充辅助文档。

可以稍后创建的辅助文档：

1. `docs/README.md`  
   当项目文档超过 2-3 份，或有协作者需要快速进入时创建。  
   它只做导航：先读什么、当前阶段是什么、其他文档在哪里。

2. `docs/ITERATION_LOG.md`  
   当项目开始连续迭代、需要记录“做了什么、为什么做、验证结果和遗留问题”时创建。  
   它用于保留上下文，避免反复追问“当时为什么这样做”。

3. `docs/archive/`  
   当阶段性规格、候选比较、一次性计划和验收记录开始增多时创建。  
   它用于保存历史信息，同时不挤占当前阅读入口。

只有在确实需要时，再创建专项文档：

1. `docs/TECHNICAL_ARCHITECTURE.md`  
   当项目开始有明确技术实现、数据流、状态管理、服务端边界或架构演进问题时创建。  
   不要在还没验证核心体验前写成完整平台架构。

2. `docs/PRODUCT_SPEC.md`  
   当某个功能、页面或 MVP 已经准备进入实现，需要明确用户流程、状态、校验和验收标准时创建。  
   不要用它替代 Project Core。

3. `docs/SAFETY.md`  
   当项目涉及儿童、隐私、健康、金融、法律、AI 输出、公开内容或其他高风险场景时创建。  
   如果项目风险较低，可以先在 Project Core 中保留简单风险边界。

4. `docs/DESIGN_PRINCIPLES.md`  
   当产品需要稳定视觉、交互、语气、信息密度和体验气质时创建。  
   不要把它写成装饰偏好清单，应服务产品原则和用户场景。

默认节奏：

```text
Project Core 先行
  -> 最小产品或内容验证
  -> 记录迭代
  -> 必要时补充专项文档
  -> 阶段完成后归档
```
